# SPEC-UI-002: 인수 테스트 기준

> 본 문서는 SPEC-UI-002의 각 EARS 요구사항(N1~N5)이 충족되었는지 검증하기 위한 Given-When-Then 인수 시나리오와 성능·품질 게이트를 정의한다.

## 인수 시나리오

### N1. 앱 진입점 및 내비게이션 쉘

**시나리오 1: 콜드 스타트 시 홈 화면 진입**
- Given: APK/IPA가 설치되어 앱이 실행되지 않은 상태
- When: 사용자가 앱을 콜드 스타트로 실행함
- Then: 시스템은 `NavigationContainer`로 감싼 네이티브 스택을 마운트하고, 초기 라우트로 `HomeScreen`을 표시한다 (크래시 없이 첫 화면 렌더)

**시나리오 2: 화면 등록 및 스택 흐름 (Home → WorkoutSelection → Camera)**
- Given: 앱이 홈 화면에 진입한 상태
- When: 사용자가 홈 → 운동 선택 → 카메라 순으로 이동함
- Then: 시스템은 `Home`, `WorkoutSelection`, `Camera` 3개 화면을 등록된 스택 라우트로 순차 표시하며, 각 전환은 네이티브 스택 애니메이션으로 동작한다

**시나리오 3: 타입 안전 라우팅 (에지 케이스)**
- Given: `RootStackParamList`로 라우트 파라미터가 정의된 상태
- When: 잘못된 라우트 이름 또는 `Camera`에 필수 `exercise` 파라미터 없이 이동을 시도하는 코드를 작성함
- Then: 시스템은 컴파일 단계(`tsc`)에서 타입 오류로 차단하며, 런타임 미등록 라우트 크래시가 발생하지 않는다

### N2. 홈 화면 (HomeScreen)

**시나리오 4: 홈 화면 구성 요소 표시**
- Given: 앱이 홈 화면에 진입한 상태
- When: 홈 화면이 렌더링됨
- Then: 시스템은 앱 타이틀("Visual PT")과 태그라인, MVP 운동 목록(스쿼트, 데드리프트), 법적 면책 고지("의료·재활 진단 도구가 아님")를 모두 표시한다

**시나리오 5: 운동 시작 버튼 이동**
- Given: 홈 화면이 표시된 상태
- When: 사용자가 "운동 시작" 버튼을 누름
- Then: 시스템은 `WorkoutSelection` 화면으로 이동한다

### N3. 운동 선택 화면 (WorkoutSelectionScreen)

**시나리오 6: 운동 카드 목록 표시**
- Given: 사용자가 운동 선택 화면에 진입한 상태
- When: 운동 선택 화면이 렌더링됨
- Then: 시스템은 스쿼트·데드리프트를 각각 설명 텍스트와 이미지 플레이스홀더를 포함한 카드 형태로 표시한다

**시나리오 7: 운동 선택 후 카메라 이동 (파라미터 전달)**
- Given: 운동 선택 화면에 스쿼트·데드리프트 카드가 표시된 상태
- When: 사용자가 스쿼트 카드의 "시작" 버튼을 누름
- Then: 시스템은 `exercise: 'squat'`를 라우트 파라미터로 담아 `Camera` 화면으로 이동한다 (데드리프트 선택 시 `exercise: 'deadlift'`)

**시나리오 8: 유효하지 않은 종목 차단 (에지 케이스)**
- Given: 운동 선택 화면이 표시된 상태
- When: 정의된 `ExerciseType` 유니온에 없는 종목으로 카메라 이동이 시도됨
- Then: 시스템은 해당 종목을 카메라 화면으로 전달하지 않으며, 타입 단계에서 차단한다

### N4. 카메라 화면 내비게이션 통합

**시나리오 9: 라우트 파라미터 기반 자동 세션 시작**
- Given: 사용자가 운동 선택 화면에서 데드리프트를 선택해 `Camera` 화면으로 이동한 상태
- When: `CameraScreen`이 마운트됨
- Then: 시스템은 `route.params.exercise`(deadlift)를 수신하여 수동 선택 버튼 없이 자동으로 `startSession()`과 추론(`startDetection`)을 시작한다

**시나리오 10: 뒤로 가기 시 세션 정리 및 복귀**
- Given: 카메라 화면에서 세션이 진행 중인 상태
- When: 사용자가 뒤로 가기를 수행함
- Then: 시스템은 활성 세션을 종료(`endSession()`, `stopDetection()`)하고 `WorkoutSelection` 화면으로 복귀하며, 추론 루프·카메라 리소스 누수가 발생하지 않는다

**시나리오 11: 운동 종목 파라미터 부재 안전 처리 (에지 케이스)**
- Given: 운동 종목 라우트 파라미터가 없는 상태로 `Camera` 화면에 진입하는 상황
- When: `CameraScreen`이 마운트됨
- Then: 시스템은 추론 세션을 시작하지 않으며, 안전하게 이전 화면으로 복귀하거나 안내를 표시한다 (크래시 없음)

### N5. 카메라 화면 실제 카메라 연결

**시나리오 12: 실제 카메라 피드 렌더링**
- Given: 카메라 권한이 허용되고 사용 가능한 카메라 디바이스가 존재하는 상태
- When: `CameraScreen`이 마운트됨
- Then: 시스템은 기존 플레이스홀더(`"카메라 대기 중"` `<View>`) 대신 `react-native-vision-camera`의 실제 `<Camera>` 컴포넌트로 라이브 카메라 피드를 표시한다

**시나리오 13: 프레임 프로세서 → 추론 연결 및 스켈레톤 오버레이**
- Given: 라이브 카메라 피드가 표시되고 세션이 진행 중인 상태
- When: 카메라 프레임이 프레임 프로세서로 도착함
- Then: 시스템은 프레임을 `usePoseDetection` 파이프라인으로 전달하여 키포인트를 추출하고, 라이브 피드 위에 실시간 포즈 스켈레톤 오버레이(`CameraOverlay`)를 정합하여 렌더링한다

**시나리오 14: 권한 거부 / 디바이스 부재 graceful fallback (에지 케이스)**
- Given: 카메라 권한이 거부되었거나 사용 가능한 카메라 디바이스가 없는 상태
- When: 사용자가 카메라 화면에 도달함
- Then: 시스템은 크래시 없이 권한 요청 화면 또는 안내 화면을 표시하며, 프레임 프로세서·추론 루프를 시작하지 않는다

## 성능 게이트 (Pass/Fail 기준)

| 항목 | 기준 | 판정 |
|------|------|------|
| 화면 전환 블로킹 | 없음 (애니메이션 외 메인 스레드 블로킹 없음) | Pass/Fail |
| 실연결 후 스켈레톤 렌더 FPS | >= 30 | Pass/Fail |
| 실연결 후 프레임당 추론 지연 | < 100ms | Pass/Fail |
| 실연결 후 피드백 응답 (감지→표시) | < 500ms | Pass/Fail |
| 화면 언마운트 후 리소스 누수 | 없음 (카메라·추론 정리 확인) | Pass/Fail |

> 성능 게이트는 SPEC-UI-001의 게이트를 **실제 카메라 연결 상태**에서 재검증하는 것을 목표로 한다. 통합 프로파일링(시나리오 9 + 12 + 13 동시 동작)으로 MVP 최소 사양 기기에서 측정한다.

|------|------|
| 테스트 커버리지 | >= 85% |
| TypeScript 타입 오류 | 0개 (`RootStackParamList` 타입 안전 라우팅 포함) |

## Definition of Done

- N1~N5 전체 EARS 요구사항에 대응하는 인수 시나리오(1~14)가 모두 Pass
- 에지 케이스(시나리오 3, 8, 11, 14) 검증 완료
- 앱이 콜드 스타트로 정상 실행되어 Home → WorkoutSelection → Camera 흐름이 동작 (설치 즉시 실행 가능)
- `CameraScreen` 플레이스홀더가 실제 `<Camera>` 컴포넌트로 교체되고 프레임 프로세서가 추론 파이프라인에 연결됨
- 라우트 파라미터 기반 자동 세션 시작 및 뒤로 가기 시 세션·카메라 정리 확인
- 성능 게이트 전 항목 통과 (실제 카메라 연결 상태, 최소 사양 기기 기준)
- 품질 게이트 전 항목 통과 (기존 테스트 무회귀 포함)
- 온디바이스 추론으로 영상·키포인트 외부 미전송 유지
- 홈·카메라 화면에 2D 카메라 한계 면책 고지 포함
