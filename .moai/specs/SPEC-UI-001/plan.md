## 1. 추천 SPEC ID 및 근거

### 추천: `SPEC-UI-001` — 실시간 자세 추정 및 폼 교정 코어

**도메인 선택 근거 (UI):**
이 SPEC의 본질은 카메라 화면(`CameraScreen.tsx`) 위에서 실시간으로 작동하는 사용자 대면 파이프라인이다. 포즈 추정, 각도 분석, 스켈레톤 오버레이, 피드백 표시, 카메라 가이드가 모두 단일 화면의 프레임 처리 루프 안에서 결합되어 동작한다. 비록 내부에 ML/수학 로직이 있으나, 사용자가 체감하는 산출물과 통합 지점이 모두 화면 렌더링 계층이므로 `UI` 도메인이 가장 적합하다.

**왜 단일 SPEC으로 묶는가 (1~4 기능 통합):**
- 카메라 → 포즈 추정 → 관절 각도 → 오류 감지 → 피드백 → 배치 가이드는 **동일한 프레임 처리 파이프라인**의 순차 단계이다. 인위적으로 분리하면 실시간 데이터 흐름이 파편화된다.
- 네 기능 모두 **단일 성능 예산**(30 FPS, 프레임당 <100ms)을 공유한다. 분리하면 성능 검증을 통합적으로 수행할 수 없다.
- 모두 `CameraScreen.tsx` 한 화면에 렌더링되며 동일한 디바이스 자원(카메라, GPU, 메인 스레드)을 경쟁한다.

**왜 세션 녹화는 분리하는가:**
- 녹화/저장/재생은 **영속성·스토리지 관심사**로 실시간 추론 루프와 결합도가 낮다.
- 안정화된 포즈 파이프라인을 기반으로 독립 개발 가능하다.
- 백엔드 업로드, 로컬 캐시, 분석 보고서 등 별도 데이터 계층을 가진다.

## 3. EARS 요구사항 구조 설계 (SPEC-UI-001, 최대 5개 모듈)

### M1. 포즈 추정 파이프라인 (Pose Estimation)
- **Ubiquitous:** 시스템은 **항상** 활성화된 운동 세션 동안 MediaPipe Pose로 33개 관절 키포인트를 온디바이스에서 추출해야 한다.
- **Event-Driven:** **WHEN** 카메라 프레임이 도착하면 **THEN** 시스템은 단일 프레임에 대한 키포인트 추론을 100ms 이내에 완료해야 한다.
- **Unwanted (UF):** **IF** 추론 신뢰도(visibility)가 임계값 미만인 키포인트가 존재하면 **THEN** 시스템은 해당 관절을 각도 계산에 사용하지 **않아야 한다**.

### M2. 관절 각도 분석 (Joint Angle Analysis)
- **Ubiquitous:** 시스템은 **항상** 키포인트로부터 무릎·엉덩이·척추·어깨 각도를 벡터 연산으로 계산해야 한다.
- **State-Driven:** **IF** 선택된 운동이 스쿼트이고 무릎 각도가 임계 범위(예: 80~120도)를 벗어나면 **THEN** 시스템은 폼 오류로 분류해야 한다.
- **State-Driven:** **IF** 선택된 운동이 데드리프트이고 척추 각도가 중립 정렬 임계를 벗어나면 **THEN** 시스템은 폼 오류로 분류해야 한다.

### M3. 실시간 피드백 시스템 (Real-time Feedback)
- **Event-Driven:** **WHEN** 폼 오류가 감지되면 **THEN** 시스템은 카메라 위에 텍스트 메시지(예: "무릎을 더 펴세요")를 표시해야 한다.
- **Event-Driven:** **WHEN** 폼 오류가 감지되면 **THEN** 시스템은 음성 피드백을 재생해야 한다.
- **Unwanted (UF):** **IF** 동일 오류가 직전 피드백 후 디바운스 구간(예: 2초) 내에 다시 감지되면 **THEN** 시스템은 중복 음성을 재생하지 **않아야 한다**.

### M4. 스켈레톤 오버레이 (Skeleton Overlay)
- **State-Driven:** **WHILE** 운동 세션이 진행 중인 동안 시스템은 카메라 피드 위에 인체 골격을 30 FPS 이상으로 렌더링해야 한다.
- **State-Driven:** **WHILE** 폼 오류가 활성 상태인 동안 시스템은 문제 관절을 경고 색상으로 강조 표시해야 한다.

### M5. 카메라 배치 가이드 (Camera Placement Guide)
- **Event-Driven:** **WHEN** 사용자가 운동 화면에 진입하면 **THEN** 시스템은 최적 카메라 위치(측면 45도, 골반 높이) 오버레이 가이드를 표시해야 한다.
- **State-Driven:** **IF** 추정된 카메라 앵글이 분석 정확도 허용 범위를 벗어나면 **THEN** 시스템은 재배치 경고 메시지를 표시해야 한다.
- **Optional:** **가능하면** 시스템은 키포인트 가시성 점수를 기반으로 자동 앵글 적정성 점수를 제공한다.

## 4. 기술 스택 결정 (버전 명시)

> 아래 버전은 SPEC 단계 권장값이며, 프로덕션 안정 버전만 포함한다. 정확한 버전 고정은 `/moai:2-run` 단계에서 최종 확정한다.

| 영역 | 라이브러리 | 권장 버전 | 역할 |
|------|-----------|----------|------|
| 프레임워크 | react-native | 0.74+ (New Architecture) | 코어 모바일 프레임워크 |
| 언어 | typescript | 5.3+ | 타입 안전성 |
| 포즈 추정 | @mediapipe/tasks-vision | 0.10+ | 온디바이스 33관절 추론 |
| 모델 | pose_landmarker_lite.tflite | (3MB) | MVP 저지연 모델 (full은 추후) |
| 카메라 | expo-camera | 15.x | 카메라 스트림 접근 |
| 오버레이 | react-native-svg | 15.x | 스켈레톤/가이드 벡터 렌더링 |
| 애니메이션 | react-native-reanimated | 3.x | 60 FPS 오버레이 갱신 (UI 스레드) |
| 워크릿/프레임 | react-native-worklets-core | 1.x | 프레임 프로세서 워크릿 실행 |
| 음성 | expo-speech | 12.x | TTS 음성 피드백 (MVP) |
| 상태 관리 | zustand | 4.x | workoutStore / feedbackStore |

**아키텍처 패턴 (실시간 카메라 처리):**
- **프레임 프로세서 패턴:** 카메라 프레임을 JS 스레드 외부(네이티브 워크릿)에서 처리하여 메인 스레드 블로킹 방지.
- **스로틀링/샘플링:** 모든 프레임이 아니라 N프레임마다 추론(예: 2프레임당 1회)하여 디바이스 부하 조절.
- **오버레이 분리 렌더링:** SVG 스켈레톤은 Reanimated 공유값(shared value)으로 UI 스레드에서 갱신, JS 리렌더 최소화.
- **임계값 설정 외부화:** 운동별 각도 임계값은 `pose.config.ts` / `feedback.config.ts`로 분리하여 운동 추가 시 확장 용이.

**성능 제약:**
- 카메라 프레임레이트: 최소 30 FPS
- 프레임당 추론 지연: <100ms
- 피드백 응답: <500ms
- 앱 메모리: <150MB
- 시간당 배터리 영향: <5%

## 5. 구현 태스크 분해 (순서 기반)

> 시간 추정 없이 우선순위·의존성 기반으로 정렬.

**1차 목표 (Primary — 파이프라인 골격):**
1. 카메라 스트림 셋업 + 권한 처리 (`useCamera`, `CameraScreen`)
2. MediaPipe tasks-vision 통합 + 프레임 프로세서 연결 (`PoseEstimationService`, `usePoseDetection`)
3. 키포인트 타입 정의 및 가시성 필터링 (`pose.types.ts`)

**2차 목표 (Secondary — 분석 로직):**
4. 관절 각도 계산 유틸 구현 (`JointAngleCalculator`, `mathHelpers`)
5. 운동별 임계값 기반 오류 감지 (`FeedbackGenerator`, `feedback.config.ts`)
6. 스쿼트/데드리프트 임계 규칙 정의 (`pose.config.ts`)

**3차 목표 (Tertiary — 피드백 출력):**
7. 스켈레톤 SVG 오버레이 렌더링 (`CameraOverlay`)
8. 텍스트 피드백 배지 + 오류 관절 강조 (`FeedbackBadge`, `JointAngleDisplay`)
9. 음성 피드백 + 디바운스 (`AudioFeedbackService`, `useFeedback`)

**최종 목표 (Final — 가이드 및 통합):**
10. 카메라 배치 가이드 오버레이 (`CameraGuideOverlay`)
11. 앵글 적정성 경고 로직
12. 통합 성능 프로파일링 (FPS/지연/메모리 측정)

## 8. 참고: 유사 React Native + ML 프로젝트 패턴

- **VisionCamera + Frame Processor 패턴:** react-native-vision-camera의 프레임 프로세서는 ML 추론을 JS 스레드 밖에서 실행하는 사실상 표준 패턴. expo-camera로 부족할 경우 대안 후보.
- **Reanimated Shared Value 오버레이:** 포즈 스켈레톤을 매 프레임 리렌더 없이 UI 스레드 공유값으로 그리는 것이 일반적 고성능 패턴.
- **MediaPipe Tasks Vision:** Google 공식 가이드는 `PoseLandmarker`를 LIVE_STREAM 모드로 실행하고 콜백으로 결과 수신 권장.
- **임계값 기반 폼 분류:** 학술/오픈소스 운동 분석 프로젝트들은 3점 벡터 각도(코사인 법칙)와 운동별 임계 범위 테이블을 사용하는 것이 정착된 접근.

> 위 패턴들은 `/moai:2-run` 단계에서 WebSearch/Context7로 최신 버전 및 공식 문서를 재검증할 것.
